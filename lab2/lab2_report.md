# Звітна записка до лабораторної роботи №2

## Тема
Реалізація каунтера в PostgreSQL та порівняння підходів до обробки одночасного звернення до БД.

## Умова тестування
1. Фіксоване очікуване значення в кожному сценарії: `100000`.
2. Сценарії навантаження:
- `1` потік × `100000` ітерацій
- `5` потоків × `20000` ітерацій
- `10` потоків × `10000` ітерацій
- `20` потоків × `5000` ітерацій

## Чотири підходи до одночасного доступу
1. `Lost-update` (читання -> зміна в коді -> запис)  
Підхід без захисту від гонок. Коли кілька потоків читають одне й те саме значення та перезаписують одне одного, частина інкрементів губиться.

2. `In-place update` (`UPDATE ... SET Counter = Counter + 1`)  
Інкремент виконується безпосередньо в SQL-операторі як атомарна дія на боці БД. Дає коректний результат при конкурентному доступі.

3. `Row-level locking` (`SELECT ... FOR UPDATE`)  
Рядок блокується на час транзакції, тому інші потоки чекають і не можуть паралельно змінювати цей самий запис. Коректно, але повільніше через очікування блокувань.

4. `Optimistic concurrency control` (через поле `Version`)  
Оновлення проходить лише якщо версія рядка не змінилась. Якщо змінилась, потік повторює спробу. Коректно, але під великим навантаженням виникає багато повторів, що суттєво збільшує час.

## Загальні висновки
1. Вимога щодо кінцевого значення `100000` виконується в усіх варіантах, крім підходу `Lost-update` при реальній конкуренції потоків.
2. `Lost-update` демонструє типову проблему одночасного доступу: втрачені оновлення.
3. `In-place update` показав найкращий баланс коректності та швидкодії для задачі простого інкременту.
4. `Row-level locking` також дає коректний результат, але працює повільніше через блокування.
5. `Optimistic concurrency control` коректний, проте на великому навантаженні стає найдовшим через численні повторні спроби оновлення.
